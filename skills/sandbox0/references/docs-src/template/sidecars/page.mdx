# Template Sidecars

Use `spec.sidecars` when a template needs extra containers in the same pod as the main sandbox container.

This is the right model for template-owned helper processes such as:

- a long-running Codex agent
- a language server
- a local proxy or daemon
- a cache warmer or bootstrap helper

All containers in one template pod share the same network namespace. When a sidecar also needs access to pod-shared storage, declare `spec.sharedVolumes` on the template and reference them from `sidecars[*].mounts`.

## Public API Shape

The public HTTP API and all three SDKs expose the same sidecar schema under `spec.sidecars`:

| Field | Required | Description |
|-------|----------|-------------|
| `name` | yes | Sidecar container name. |
| `image` | yes | Container image reference. |
| `command` | no | Entrypoint override. |
| `args` | no | Additional process arguments. |
| `env` | no | Sidecar-specific environment variables. |
| `resources` | yes | CPU and memory request/limit pair used by Sandbox0's template schema. |
| `mounts` | no | Shared-volume mount references for this sidecar. |
| `readinessProbe` | no | Readiness probe. Exactly one handler is required when set. It controls when the sidecar is considered ready for warm-pool capacity and hot claims. |
| `startupProbe` | no | Startup probe. Exactly one handler is required when set. |

<Callout variant="warning">
`livenessProbe` is intentionally unsupported for template sidecars. Sandbox0 soft pause can freeze the whole sandbox, so kubelet liveness restarts would conflict with pause/resume.
</Callout>

If a sidecar needs pod-shared storage, define it once under `spec.sharedVolumes`:

| Field | Required | Description |
|-------|----------|-------------|
| `name` | yes | Logical shared-volume name used by `sidecars[*].mounts[*].name`. |
| `sandboxVolumeId` | no | Optional fixed SandboxVolume ID. Omit it when the volume should be bound at claim time. |
| `mountPath` | yes | Mount path used by the main container. |
| `cacheSize` | no | Optional cache size override. |
| `prefetch` | no | Optional prefetch override. |
| `bufferSize` | no | Optional buffer-size override. |
| `writeback` | no | Optional writeback flag. |

<Callout variant="info">
`sharedVolumes` only matters when more than one container in the template needs the same mounted volume. If `sandboxVolumeId` is omitted, claim-time mounts bind the shared volume by matching `sharedVolumes[*].mountPath` to `mounts[*].mount_point` in the sandbox claim request.
</Callout>

<Callout variant="info">
If your template logic lives entirely in a sidecar and the main container does not need to run interactive commands or REPL sessions, you can keep `mainContainer.image` minimal, for example `busybox:1.36`. Avoid floating tags such as `busybox:latest`, otherwise each refresh can trigger unexpected image pulls.
</Callout>

## Shared Volumes For Sidecars

`spec.sharedVolumes` is template-level. Each entry defines one pod-shared SandboxVolume attachment.

- The main container sees the shared volume at `sharedVolumes[*].mountPath`.
- A sidecar sees the same volume only when it adds a matching `mounts[*].name` entry.
- Each sidecar can choose a different `mounts[*].mountPath` for the same shared volume.

Example:

```yaml
spec:
    mainContainer:
        image: ubuntu:24.04
        resources:
            cpu: "1"
            memory: 4Gi
    sharedVolumes:
        - name: workspace
          # sandboxVolumeId: vol_123  # optional
          mountPath: /workspace/shared
          writeback: true
    sidecars:
        - name: codex
          image: ghcr.io/example/codex:v1
          resources:
              cpu: "250m"
              memory: 1Gi
          command: ["sh", "-lc", "/app/start-codex.sh"]
          mounts:
              - name: workspace
                mountPath: /shared
```

In this example, the main container uses `/workspace/shared` while the sidecar uses `/shared`, but both paths point at the same SandboxVolume.

## Sandbox0 Readiness Controls Claimability

Warm-pool capacity only counts pods that are ready for claiming.

- A pod in `Running` phase with `Ready=False` does not count as available idle capacity.
- A hot claim only selects idle pods that are already ready.
- Sidecar `readinessProbe` can delay when a pod becomes claimable.

Example:

```yaml
spec:
    mainContainer:
        image: ubuntu:24.04
        resources:
            cpu: "1"
            memory: 4Gi
    sharedVolumes:
        - name: workspace
          # sandboxVolumeId: vol_123  # optional
          mountPath: /workspace/shared
    sidecars:
        - name: codex
          image: ghcr.io/example/codex:v1
          resources:
              cpu: "250m"
              memory: 1Gi
          command: ["sh", "-lc", "/app/start-codex.sh"]
          mounts:
              - name: workspace
                mountPath: /shared
          startupProbe:
              exec:
                  command: ["test", "-f", "/tmp/codex-started"]
    pool:
        minIdle: 2
        maxIdle: 10
```

In this example, the sidecar can model slow startup with `startupProbe`, but the pod is not counted as ready warm capacity until it becomes claimable.

## Validation Notes

- `name`, `image`, and `resources` are required for every sidecar.
- `mounts[*].name` must refer to a declared `sharedVolumes[*].name`.
- `startupProbe` must define exactly one handler when present.
- Private image references are still checked against team ownership rules, just like `spec.mainContainer.image`.
