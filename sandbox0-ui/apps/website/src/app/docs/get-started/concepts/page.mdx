# Core Concepts

Understanding the core concepts of Sandbox0 will help you build effective AI Agent applications. This page covers the fundamental building blocks.

## Sandbox

A **Sandbox** is an isolated execution environment where you can run code, manage files, and control network access. Each sandbox is created from a [Template](#template) and provides a secure, ephemeral workspace.

### Lifecycle States

Sandboxes progress through defined lifecycle states:

| Status | Description | Transitions |
|--------|-------------|-------------|
| `starting` | Sandbox is being initialized | → `running` or `failed` |
| `running` | Sandbox is active and ready to use | → `paused`, `completed`, or `failed` |
| `paused` | Sandbox is paused (resources preserved) | → `running` or `completed` |
| `failed` | Sandbox encountered an error | → `completed` |
| `completed` | Sandbox has finished execution | Terminal state |

### Key Properties

| Property | Description |
|----------|-------------|
| `id` | Unique identifier (e.g., `sb_abc123`) |
| `template_id` | Template used to create this sandbox |
| `status` | Current lifecycle state |
| `ttl` | Time-to-live in seconds |
| `hard_ttl` | Maximum TTL (cannot be extended beyond this) |
| `expires_at` | When the sandbox will be automatically terminated |

### TTL Management

- **TTL (Time-to-Live)**: Default lifetime of the sandbox
- **Hard TTL**: Maximum lifetime that cannot be exceeded
- **Refresh**: Extend the TTL up to the hard TTL limit
- **Auto-resume**: Automatically resume a paused sandbox when accessed

<Callout variant="info">
Sandboxes are designed for ephemeral workloads. For persistent data storage, use [Volumes](#volume).
</Callout>

## Context (Process Runtime)

A **Context** represents a process running inside a sandbox. Sandbox0 supports two context types for different execution patterns.

### Context Types

| Type | Description | Use Case |
|------|-------------|----------|
| **REPL** | Read-Eval-Print Loop (interactive) | Python interpreter, Node.js REPL, language shells |
| **Cmd** | One-shot command execution | Shell scripts, single commands, batch processing |

### REPL Contexts

REPL contexts maintain state across multiple inputs:

<Tabs
  tabs={[
    {
      label: "Go",
      language: "go",
      code: `// Create a Python REPL context
processType := sandbox0.REPL
ctxResp, err := client.CreateContext(ctx, sandbox.ID, &sandbox0.CreateContextRequest{
    Type: &processType,
    REPL: &sandbox0.CreateREPLContextRequest{
        Command: []string{"python3", "-i"},
    },
})

// Send input and receive output
input := "x = 1 + 2\\nprint(x)"
output, err := client.ExecContext(ctx, sandbox.ID, ctxResp.ID, input)
fmt.Println(output.Stdout) // Output: 3

// State persists - x is still defined
input = "print(x * 2)"
output, err = client.ExecContext(ctx, sandbox.ID, ctxResp.ID, input)
fmt.Println(output.Stdout) // Output: 6`
    },
    {
      label: "Python",
      language: "python",
      code: `# Create a Python REPL context
context = client.create_context(
    sandbox.id,
    context_type="repl",
    command=["python3", "-i"],
)

# Send input and receive output
output = client.exec_context(sandbox.id, context.id, "x = 1 + 2\\nprint(x)")
print(output.stdout)  # Output: 3

# State persists
output = client.exec_context(sandbox.id, context.id, "print(x * 2)")
print(output.stdout)  # Output: 6`
    },
    {
      label: "TypeScript",
      language: "typescript",

    type: 'repl',

});

console.log(output.stdout); // Output: 3

// State persists

console.log(output.stdout); // Output: 6`
    },
    {
      label: "CLI",
      language: "bash",
      code: `# Create and interact with a REPL context

# Context ID: ctx_xyz789

# Send input

# Output: 3

# State persists
s0 context exec sb_abc123 ctx_xyz789 --input "print(x * 2)"
# Output: 6`
    }
  ]}

### Cmd Contexts

Cmd contexts execute a single command and complete:

<Tabs
  tabs={[
    {
      label: "Go",
      language: "go",
      code: `// Create a command context
processType := sandbox0.Cmd

    Type: &processType,
    Cmd: &sandbox0.CreateCMDContextRequest{
        Command: []string{"/bin/sh", "-c", "echo 'Hello' && sleep 1 && echo 'World'"},
    },
})

// Wait for completion
status, err := client.GetContext(ctx, sandbox.ID, ctxResp.ID)
fmt.Printf("Exit code: %d\\n", status.ExitCode)`
    },
    {
      label: "Python",
      language: "python",
      code: `# Create a command context

    sandbox.id,
    context_type="cmd",
    command=["/bin/sh", "-c", "echo 'Hello' && sleep 1 && echo 'World'"],
)

# Wait for completion
status = client.get_context(sandbox.id, context.id)
print(f"Exit code: {status.exit_code}")`
    },
    {
      label: "TypeScript",
      language: "typescript",
      code: `// Create a command context

// Wait for completion

### WebSocket Streaming

For real-time interaction, connect to a context via WebSocket:

```
wss://api.sandbox0.ai/api/v1/sandboxes/{sandbox_id}/contexts/{context_id}/ws
```

WebSocket message types:
- **Input**: `{"type": "input", "data": "command\n"}`
- **Output**: `{"type": "output", "source": "stdout", "data": "result\n"}`
- **Resize**: `{"type": "resize", "rows": 24, "cols": 80}`
- **Signal**: `{"type": "signal", "signal": "INT"}`

## Template

A **Template** defines the blueprint for creating sandboxes. It specifies the container image, resource requirements, pool configuration, and other settings.

### Template Components

| Component | Description |
|-----------|-------------|
| `mainContainer` | Primary container specification (image, resources, env) |
| `sidecars` | Additional containers running alongside the main container |
| `pool` | Pre-warmed sandbox pool configuration |
| `lifecycle` | Startup probes and lifecycle hooks |
| `network` | Default network policy |

### Pool Configuration

Templates support pool-based provisioning for fast cold starts:

| Field | Description |
|-------|-------------|
| `minIdle` | Minimum number of pre-warmed sandboxes |
| `maxIdle` | Maximum number of idle sandboxes |
| `autoScale` | Enable automatic scaling |

<Callout variant="success">
Pre-warmed pools enable sub-200ms cold starts. When you claim a sandbox, it's immediately available from the pool.
</Callout>

### Template Visibility

| Scope | Description |
|-------|-------------|
| `public` | Visible to all users in the organization |
| `private` | Only visible to the owning team |
| `team-restricted` | Visible only to specified teams |

## Volume

A **Volume** provides persistent storage that can be mounted to sandboxes. Unlike sandbox-local storage, volumes survive sandbox termination.

### Volume Lifecycle

| Status | Description |
|--------|-------------|
| `available` | Volume is ready to be mounted |
| `mounted` | Volume is currently mounted to a sandbox |
| `error` | Volume encountered an error |

### Access Modes

| Mode | Description | Use Case |
|------|-------------|----------|
| **RWO** | ReadWriteOnce | Single sandbox read/write |
| **ROX** | ReadOnlyMany | Multiple sandboxes read-only |
| **RWX** | ReadWriteMany | Multiple sandboxes read/write |

### Mount Points

Mount a volume to a sandbox at a specified path:

<Tabs

### Snapshots

Create point-in-time snapshots of volumes for backup or cloning:

| Operation | Description |
|-----------|-------------|
| Create snapshot | Capture current volume state |
| List snapshots | View all snapshots for a volume |
| Restore snapshot | Restore volume to a previous state |
| Delete snapshot | Remove a snapshot |

## Network Policy

**Network Policy** controls the network access of a sandbox. You can restrict outbound traffic to specific domains or block all external access.

### Policy Modes

| Mode | Description |
|------|-------------|
| `BlockAll` | Block all external network access |
| `AllowAll` | Allow all external network access |
| `Custom` | Fine-grained rules for allowed/blocked domains |

### Custom Rules
