# Volume Fork

Volume Fork creates a new Volume from an existing Volume using JuiceFS Copy-on-Write (COW) clone.

## Why Use Fork?

- **Fast Clone**: Create a new volume quickly without copying all data blocks
- **Storage Efficient**: Source and fork share unchanged data blocks
- **Write Isolation**: Writes to the fork do not affect the source volume
- **Great for Branching**: Create per-task or per-experiment data branches

## How Fork Works

```
Source Volume (vol_src)
    ├── /data/a.txt
    └── /data/b.txt
            │
            │ Fork (COW clone)
            ▼
Forked Volume (vol_fork)
    ├── /data/a.txt  (shared blocks)
    └── /data/b.txt  (shared blocks)

After writing in fork:
    vol_fork /data/b.txt -> new blocks
    vol_src  /data/b.txt -> unchanged
```

## API

- **Endpoint**: `POST /api/v1/sandboxvolumes/{id}/fork`
- **Path Param**: `id` = source volume id
- **Request Body**: optional
- **Response**: returns the new `SandboxVolume`, including `source_volume_id`

## Optional Request Fields

| Field | Type | Description |
|---|---|---|
| `cache_size` | string | Cache size for the forked volume |
| `prefetch` | integer | Prefetch blocks |
| `buffer_size` | string | Buffer size |
| `writeback` | boolean | Writeback cache switch |
| `access_mode` | enum | Access mode for the forked volume, defaults to `RWO` |

## Verify Isolation

1. Mount both source and forked volumes to a sandbox at different paths.
2. Modify a file only in the forked volume.
3. Read the same file from source volume and verify it is unchanged.

<Callout variant="info">
Fork uses COW at filesystem metadata layer, so unchanged data is shared and changed data is isolated.
</Callout>

## Best Practices

- Use fork for short-lived branches (feature experiments, data transformations, CI tasks)
- Keep a clear naming convention for source/fork relationship
- Clean up unused forks to reduce metadata and management overhead
