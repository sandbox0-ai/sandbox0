# Troubleshooting

Common issues and solutions when deploying and operating sandbox0.

## Installation Issues

### Sandbox0Infra Stuck in "Installing" Phase

**Symptoms:**
- `kubectl get sandbox0infra` shows `Phase: Installing` for extended time
- `Progress` field shows incomplete count (e.g., "3/8")

**Diagnosis:**

<Tabs
  tabs={[
    {
      label: "CLI",
      language: "bash",
      code: `# Check detailed status and conditions
kubectl describe sandbox0infra <name> -n sandbox0-system

# Check operator logs
kubectl logs -n sandbox0-system -l app.kubernetes.io/name=infra-operator

# Check specific component pods
kubectl get pods -n sandbox0-system
kubectl describe pod <pod-name> -n sandbox0-system`,
      mode: "run",
      needs: "k8s",
      session: "selfhosted-troubleshoot-installing-diagnosis",
      setup: ["command -v kubectl >/dev/null"]
    }
  ]}
/>

**Common Causes:**

1. **Image Pull Failures**
   <Tabs
     tabs={[
       {
         label: "CLI",
         language: "bash",

# Verify image pull secrets
kubectl get secrets -n sandbox0-system | grep pull`,
         mode: "run",
         needs: "k8s",
         session: "selfhosted-troubleshoot-imagepull",
         setup: ["command -v kubectl >/dev/null"]
       }
     ]}
   />

2. **PVC Pending**
   <Tabs
     tabs={[
       {
         label: "CLI",
         language: "bash",
         code: `# Check PVC status

# Verify storage class exists
kubectl get storageclass`,
         mode: "run",
         needs: "k8s",
         session: "selfhosted-troubleshoot-pvc",
         setup: ["command -v kubectl >/dev/null"]

     ]}

3. **Resource Constraints**
   <Tabs
     tabs={[

### Database Connection Failures

**Symptoms:**
- Pods fail health checks
- Logs show connection refused or timeout

**For Built-in Database:**

<Tabs
  tabs={[
    {
      label: "CLI",
      language: "bash",
      code: `# Check database pod status
kubectl get pods -n sandbox0-system -l app.kubernetes.io/component=database

# Check database logs
kubectl logs -n sandbox0-system -l app.kubernetes.io/component=database

# Verify database is ready
kubectl exec -n sandbox0-system <database-pod> -- pg_isready`,
      mode: "run",
      needs: "k8s",
      session: "selfhosted-troubleshoot-db-builtin",
      setup: ["command -v kubectl >/dev/null"]
    }
  ]}
/>

**For External Database:**

<Tabs
  tabs={[
    {
      label: "CLI",
      language: "bash",
      code: `# Verify credentials secret exists
kubectl get secret <secret-name> -n sandbox0-system

# Test connectivity from a test pod
kubectl run pg-test --rm -it --image=postgres:16-alpine -- \
  psql "postgresql://<username>:<password>@<host>:<port>/<database>"`,
      mode: "run",
      needs: "k8s",
      session: "selfhosted-troubleshoot-db-external",
      setup: ["command -v kubectl >/dev/null"]
    }
  ]}
/>

**Common Fixes:**

1. Verify the password secret has the correct key name
2. Check network policies allow egress to the database
3. Verify SSL mode matches database configuration

### FUSE-Related Errors

**Symptoms:**
- Storage-proxy logs show FUSE mount failures
- Error: "fusermount: mount failed: permission denied"

**Solution:**

Ensure nodes have FUSE available and the storage-proxy pod has appropriate capabilities:

<Tabs
  tabs={[
    {
      label: "CLI",
      language: "bash",
      code: `# Check if FUSE is available on nodes
kubectl debug node/<node-name> -it --image=alpine -- ls /dev/fuse

# Verify storage-proxy has necessary capabilities
kubectl get pod <storage-proxy-pod> -n sandbox0-system -o yaml | grep -A 5 capabilities`,
      mode: "run",
      needs: "k8s",
      session: "selfhosted-troubleshoot-fuse",
      setup: ["command -v kubectl >/dev/null"]
    }
  ]}
/>

<Callout variant="info">
For Kubernetes environments without FUSE support, you may need to use a CSI driver or privileged mode for storage-proxy.
</Callout>

## Network Policy Issues

### netd Not Applying Policies

**Symptoms:**
- Network policies defined but not enforced
- Sandbox pods can still access blocked domains

**Diagnosis:**

<Tabs
  tabs={[
    {
      label: "CLI",
      language: "bash",
      code: `# Check netd DaemonSet
kubectl get daemonset -n sandbox0-system -l app.kubernetes.io/component=netd

# Check netd logs on specific node
kubectl logs -n sandbox0-system <netd-pod>

# Verify pod annotations
kubectl get pod <sandbox-pod> -n <namespace> -o jsonpath='{.metadata.annotations}'`,
      mode: "run",
      needs: "k8s",
      session: "selfhosted-troubleshoot-netd-policy",
      setup: ["command -v kubectl >/dev/null"]
    }
  ]}
/>

**Common Fixes:**

1. Ensure netd runs with hostNetwork and appropriate capabilities
2. Verify iptables is accessible from netd pods
3. Check for conflicts with other network plugins

## Service Communication Issues

### Internal Authentication Failures

**Symptoms:**
- 401/403 errors between services
- Logs show "invalid token" or "signature verification failed"

**Diagnosis:**

<Tabs
  tabs={[
    {
      label: "CLI",
      language: "bash",
      code: `# Check internal auth secrets
kubectl get secrets -n sandbox0-system | grep auth

# Verify public key is accessible
kubectl get secret <public-key-secret> -n sandbox0-system -o jsonpath='{.data.public\.key}' | base64 -d`,
      mode: "run",
      needs: "k8s",
      session: "selfhosted-troubleshoot-internal-auth",
      setup: ["command -v kubectl >/dev/null"]
    }
  ]}
/>

**Solution:**

1. Verify the same key pair is used across services
2. For multi-cluster, ensure data plane has control plane's public key
3. Check key format (should be ed25519)

### Control Plane Connection Issues (Multi-Cluster)

**Symptoms:**
- Data plane cannot reach control plane
- Cluster registration fails

**Diagnosis:**

<Tabs
  tabs={[
    {
      label: "CLI",
      language: "bash",
      code: `# Test connectivity from data plane
kubectl run curl-test --rm -it --image=curlimages/curl -- \
  curl -v https://<control-plane-url>/health

# Check control plane configuration
kubectl get sandbox0infra -n sandbox0-system -o yaml | grep -A 5 controlPlane`,
      mode: "run",
      needs: "k8s",
      session: "selfhosted-troubleshoot-controlplane-connection",
      setup: ["command -v kubectl >/dev/null"]
    }
  ]}
/>

**Common Fixes:**

1. Verify control plane URL is accessible from data plane cluster
2. Check TLS certificates are valid
3. Ensure internal auth public key is correctly configured

## Performance Issues

### Slow Sandbox Startup

**Symptoms:**
- Sandboxes take longer than expected to become ready
- High latency on claim operations

**Diagnosis:**

<Tabs
  tabs={[
    {
      label: "CLI",
      language: "bash",
      code: `# Check manager metrics
kubectl port-forward -n sandbox0-system <manager-pod> 8080:8080 &
curl http://localhost:8080/metrics

# Check warm pool status
kubectl get sandboxtemplate -A
kubectl get replicaset -A -l sandbox0.io/template-id`,
      mode: "run",
      needs: "k8s",
      session: "selfhosted-troubleshoot-startup-performance",
      setup: ["command -v kubectl >/dev/null", "command -v curl >/dev/null"]
    }
  ]}
/>

**Common Fixes:**

1. Increase `minIdle` for templates to maintain larger warm pools
2. Use faster storage class for database and storage
3. Optimize container images (smaller size, fewer layers)

### High Resource Usage

**Symptoms:**
- Nodes running low on resources
- OOMKilled pods

**Diagnosis:**

<Tabs
  tabs={[
    {
      label: "CLI",
      language: "bash",
      code: `# Check resource usage
kubectl top pods -n sandbox0-system
kubectl top nodes

# Check resource limits
kubectl describe pod <pod-name> -n sandbox0-system | grep -A 5 "Limits:"`,
      mode: "run",
      needs: "k8s",
      session: "selfhosted-troubleshoot-high-resource",
      setup: ["command -v kubectl >/dev/null"]
    }
  ]}
/>

**Solution:**

Adjust resource requests and limits in the services configuration:

<Tabs
  tabs={[
    {
      label: "YAML",
      language: "yaml",
      code: `spec:
  services:
    manager:
      resources:
        requests:
          cpu: "200m"
          memory: "256Mi"
        limits:
          cpu: "1000m"
          memory: "1Gi"`,
      mode: "read",
      needs: "none",
      session: "selfhosted-troubleshoot-manager-resources-yaml"
    }
  ]}
/>

## Getting Help

If you continue to experience issues:

1. **Check the logs** of the affected component
2. **Review the conditions** in `Sandbox0Infra` status
3. **Gather diagnostic information:**
   <Tabs
     tabs={[
       {
         label: "CLI",
         language: "bash",
         code: `kubectl cluster-info dump -n sandbox0-system > cluster-dump.txt`,
         mode: "run",
         needs: "k8s",
         session: "selfhosted-troubleshoot-cluster-dump",
         setup: ["command -v kubectl >/dev/null"]
       }
     ]}
   />
4. **Contact support** with the diagnostic information and a description of the issue
