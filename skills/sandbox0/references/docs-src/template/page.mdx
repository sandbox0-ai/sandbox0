# Template

A Template is the blueprint for creating Sandboxes. It defines the container image, resource limits, warm pool size, and default network policy. Every Sandbox is created from a Template.

## Template Types

Sandbox0 has two categories of templates:

| Type | Description | Visibility |
|------|-------------|------------|
| **Builtin** | System-provided templates managed by the platform operator | Public — visible to all teams |
| **Custom** | Templates created and owned by your team | Private — visible to your team only (configurable) |

Builtin templates use curated public images and are ready to use immediately. Custom templates let you bring your own image, and fine-tune resources.

Custom templates can also define sidecar containers. This is useful for template-owned warm processes, for example a long-running Codex agent or helper daemon. Pool capacity only counts pods that become Kubernetes-ready, so a sidecar readiness probe can gate whether a warm pod is claimable. See [Template Sidecars](/docs/template/sidecars) for the dedicated guide.

## Create Template

Create a custom template for your team. The template spec is defined in a YAML file.

<Endpoint method="POST">
/api/v1/templates
</Endpoint>

### Request Body

| Field | Type | Description |
|-------|------|-------------|
| `template_id` | string | Unique identifier for the template (e.g., `my-python-env`) |
| `spec` | object | Full template specification |

<Callout variant="info">
Template IDs must be unique within your team. Once created, the ID cannot be changed — create a new template if you need a different ID.
</Callout>

**Example spec file (`template.yaml`):**

```yaml
spec:
    mainContainer:
        image: python:3.12-slim
        resources:
            cpu: "1"
            memory: 2Gi
    pool:
        minIdle: 2
        maxIdle: 10
```

**Example with a sidecar-hosted warm process:**

```yaml
spec:
    mainContainer:
        image: ubuntu:24.04
        resources:
            cpu: "1"
            memory: 2Gi
    sidecars:
        - name: codex
          image: ghcr.io/example/codex:latest
          command: ["sh", "-lc", "/app/start-codex.sh"]
          readinessProbe:
            exec:
              command: ["test", "-S", "/tmp/codex.sock"]
    pool:
        minIdle: 2
        maxIdle: 10
```

Use the sidecar readiness probe to express when the warm process is truly ready. A pod that is merely `Running` but not `Ready` is not counted as available pool capacity.

## Update Template

Update the spec of an existing custom template. The update triggers a reconciliation across all clusters — running sandboxes are not affected.

<Endpoint method="PUT">
/api/v1/templates/{'{id}'}
</Endpoint>

<Callout variant="warning">
Updating a template does not affect already-running Sandboxes. New Sandboxes claimed after the update will use the new spec. Idle pods in the pre-warm pool are recycled and replaced with pods matching the new spec.
</Callout>

<Tabs
  tabs={[
    {
      label: "Go",
      language: "go",
      code: `tpl, err = client.UpdateTemplate(ctx, "my-python-env", apispec.TemplateUpdateRequest{
    Spec: apispec.SandboxTemplateSpec{
        MainContainer: apispec.NewOptContainerSpec(apispec.ContainerSpec{
            Image: "python:3.12-slim",
            Resources: apispec.ResourceQuota{
                CPU:    apispec.NewOptString("2"),
                Memory: apispec.NewOptString("4Gi"),
            },
        }),
        Pool: apispec.NewOptPoolStrategy(apispec.PoolStrategy{
            MinIdle: 3,
            MaxIdle: 10,
        }),
    },
})
if err != nil {
    log.Fatal(err)
}
fmt.Printf("Template updated: %s\\n", tpl.TemplateID)`
    },
    {
      label: "Python",
      language: "python",
      code: `from sandbox0.apispec.models.template_update_request import TemplateUpdateRequest
from sandbox0.apispec.models.sandbox_template_spec import SandboxTemplateSpec

tpl = client.update_template(
    "my-python-env",
    TemplateUpdateRequest(
        spec=SandboxTemplateSpec.from_dict({
            "mainContainer": {
                "image": "python:3.12-slim",
                "resources": {"cpu": "2", "memory": "4Gi"},
            },
            "pool": {"minIdle": 3, "maxIdle": 10},
        }),
    ),
)
print(f"Template updated: {tpl.template_id}")`
    },
    {
      label: "TypeScript",
      language: "typescript",

            image: "python:3.12-slim",
            resources: { cpu: "2", memory: "4Gi" },

        pool: { minIdle: 3, maxIdle: 10 },
    },
});
console.log("Template updated:", tpl.templateId);`

## Delete Template

Delete a custom template. All cluster allocations for the template are cleaned up via reconciliation.

<Endpoint method="DELETE">
/api/v1/templates/{'{id}'}
</Endpoint>

<Callout variant="warning">
Deleting a template does not terminate running Sandboxes created from it, but no new Sandboxes can be created from the deleted template.
</Callout>

<Tabs
  tabs={[
    {
      label: "Go",
      language: "go",
      code: `_, err = client.DeleteTemplate(ctx, "my-python-env")
if err != nil {
    log.Fatal(err)
}
fmt.Println("Template deleted")`
    },
    {
      label: "Python",
      language: "python",
      code: `client.delete_template("my-python-env")
print("Template deleted")`
    },
    {
      label: "TypeScript",

    },
    {
      label: "CLI",

    }
  ]}
