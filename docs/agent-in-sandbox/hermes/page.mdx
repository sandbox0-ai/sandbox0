# Hermes

Run Hermes inside a Sandbox0 sandbox when you want Hermes sessions, memory, skills, and gateway state to live on a SandboxVolume while Sandbox0 controls runtime lifecycle and public ingress.

The builtin `hermes` template uses the upstream Hermes image and declares `/opt/data` as the persistent mount point.

## Create The Sandbox

This flow creates a Hermes data volume, claims a sandbox from the builtin template, and exposes the API server through a `cmd` service. It leaves `ttl` and `hard_ttl` unset.

```bash
export SANDBOX0_BASE_URL="${SANDBOX0_BASE_URL:-https://api.sandbox0.ai}"
export HERMES_API_SERVER_KEY="$(openssl rand -hex 32)"
export HERMES_ROUTE_TOKEN="$(openssl rand -hex 32)"
export HERMES_ROUTE_TOKEN_SHA256="$(printf '%s' "$HERMES_ROUTE_TOKEN" | sha256sum | awk '{print $1}')"

HERMES_VOLUME_ID="$(
    curl -fsS "$SANDBOX0_BASE_URL/api/v1/sandboxvolumes" \
        -H "authorization: Bearer $SANDBOX0_TOKEN" \
        -H "content-type: application/json" \
        -d '{"access_mode":"RWO"}' \
        | jq -r '.data.id'
)"

cat > hermes-sandbox.yaml <<EOF
template: hermes
mounts:
    - sandboxvolume_id: "$HERMES_VOLUME_ID"
      mount_point: /opt/data
config:
    auto_resume: true
    env_vars:
        HERMES_HOME: /workspace/.hermes
        HERMES_PERSIST_HOME: /opt/data
        HOME: /workspace/.hermes
        API_SERVER_ENABLED: "true"
        API_SERVER_HOST: 0.0.0.0
        API_SERVER_PORT: "8642"
        API_SERVER_KEY: "$HERMES_API_SERVER_KEY"
    services:
        - id: gateway
          port: 8642
          runtime:
              type: cmd
              command:
                  - sh
                  - -lc
                  - |
                      set -eu
                      HERMES_HOME="\${HERMES_HOME:-/workspace/.hermes}"
                      HERMES_PERSIST_HOME="\${HERMES_PERSIST_HOME:-/opt/data}"
                      export HERMES_HOME HERMES_PERSIST_HOME HOME="\$HERMES_HOME"
                      mkdir -p "\$HERMES_HOME" "\$HERMES_PERSIST_HOME"
                      cp -R "\$HERMES_PERSIST_HOME"/. "\$HERMES_HOME"/ 2>/dev/null || true
                      rm -f "\$HERMES_HOME/gateway.lock" "\$HERMES_HOME/gateway.pid"
                      chown -R hermes:hermes "\$HERMES_HOME"
                      sync_back() {
                          mkdir -p "\$HERMES_PERSIST_HOME"
                          cp -R "\$HERMES_HOME"/. "\$HERMES_PERSIST_HOME"/ 2>/dev/null || true
                      }
                      /opt/hermes/bin/hermes gateway run --no-supervise &
                      child="\$!"
                      trap 'kill -TERM "\$child" 2>/dev/null || true; wait "\$child" 2>/dev/null || true; sync_back; exit 143' TERM INT
                      wait "\$child"
                      status="\$?"
                      sync_back
                      exit "\$status"
          health_check:
              path: /health
          ingress:
              public: true
              routes:
                  - id: gateway
                    path_prefix: /
                    methods: [GET, POST]
                    auth:
                        mode: bearer
                        bearer_token_sha256: "$HERMES_ROUTE_TOKEN_SHA256"
                    timeout_seconds: 60
                    resume: true
EOF

SANDBOX_ID="$(s0 -o json sandbox create -f hermes-sandbox.yaml | jq -r '.ID // .id')"
echo "$SANDBOX_ID"
```

Sandbox0 starts Hermes through `procd`, so the example runs the Hermes binary directly instead of relying on the Docker image entrypoint. Hermes keeps its active runtime home on `/workspace/.hermes` and syncs it with the persistent `/opt/data` volume when the process exits.

## Use The Public URL

Fetch the service details and call the health endpoint.

```bash
HERMES_PUBLIC_URL="$(
    s0 -o json sandbox service get -s "$SANDBOX_ID" \
        | jq -r '(.Services // .services)[] | select((.ID // .id) == "gateway") | (.PublicURL // .public_url)'
)"

curl -fsS "$HERMES_PUBLIC_URL/health" \
    -H "authorization: Bearer $HERMES_ROUTE_TOKEN"
```

Use the returned service `public_url` as `HERMES_PUBLIC_URL`. Requests that use the Hermes API server should also send the Hermes API server key according to the Hermes API server contract.

## Pause And Resume

The sandbox can be paused manually:

```bash
s0 sandbox pause "$SANDBOX_ID"

until [ "$(
    curl -fsS "$SANDBOX0_BASE_URL/api/v1/sandboxes/$SANDBOX_ID" \
        -H "authorization: Bearer $SANDBOX0_TOKEN" \
        | jq -r '.data.status'
)" = "paused" ]; do
    sleep 1
done

curl -fsS "$HERMES_PUBLIC_URL/health" \
    -H "authorization: Bearer $HERMES_ROUTE_TOKEN"
```

The request can resume the sandbox because the sandbox has `auto_resume: true` and the route has `resume: true`. Sandbox0 starts a new runtime pod and then starts the `cmd` service again. Hermes state under `/opt/data` remains on the mounted volume.

<Callout variant="warning">
Paused sandboxes cannot receive outbound long-polling, WebSocket, or cron events. Put message webhooks and schedulers in an always-on control plane that calls the Sandbox0 service URL, or keep the sandbox running for workflows that must listen continuously.
</Callout>

## Secret Boundary

Hermes can store model keys and integration secrets in its data directory. For higher-control deployments, keep provider credentials outside the sandbox and route model calls through LLMProxy, egress auth, or another narrow external proxy. The mounted `/opt/data` volume should hold Hermes state, not broad platform credentials.

## Operator Notes

The builtin template is configurable through `Sandbox0Infra.spec.builtinTemplates`. Remove `templateId: hermes` from that list to delete the operator-managed public template, or replace its `image`, `pool`, or full `spec` to pin a reviewed Hermes image.
